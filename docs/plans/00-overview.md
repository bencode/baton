## Context（为什么做）

现状里已有 tcollab(Web-first 的 Claude Code 协作平台:entry-repo 编排 + ttyd 会话),
以及参照系 Helm/helmagent.dev(常驻 coding agent daemon + Kanban + Planner/Coder 等人格)。
两者都把"一个 agent 替人干活"做成了产品,但都**不具备**:多渠道入口、显式的"需求"组织层、
以及"端注册能力、按能力消费任务"的协同模型。

baton 要解决的是**协同**:让需求和任务在一个工作空间内共享,多个能力各异的执行端(worker)
自注册并接活,把"单 agent 干活"升级成"共享任务池 + 多 worker 协同"。
"baton(接力棒)"= 工作在节点间交接的隐喻。

baton 是**全新独立项目**(空仓库 `git@github.com:bencode/baton.git`),
不依赖也不复用 tcollab,只借鉴思路。

## 愿景一句话

一个 **agent 协同执行引擎**:工作空间 → 需求 → 任务三层共享状态住在常驻 daemon 里,
人和 agent 通过多种双向渠道(飞书/钉钉/CLI)创建需求与任务,
注册了能力的 worker 经长连接接活、协同把任务跑完。

## 领域模型

```
Workspace (工作空间)        共享边界。其下需求/任务/worker 注册对空间内可见
   └── Requirement (需求)   聚合根:上下文容器 + 一组任务。本身不执行,状态派生
          └── Task (任务)   执行单元:有状态机、能力要求、依赖(depends_on)
Worker (执行端)             注册能力,被动接受引擎 push 的任务;可以是 agent / 脚本 / 人
```

- **Workspace** — 协同的边界与隔离单位。
- **Requirement** — **不承载执行**,只持有:描述、资源引用(文档/链接/文件)、特性标签。
  状态是**派生的**(下属任务全完成 → 需求完成)。是组织维度 + 上下文提供者。
- **Task** — 真正的执行单元。有自己的状态机、**能力要求**(用于匹配 worker,
  可从所属需求的标签/资源继承部分上下文)、以及 `depends_on` 依赖(形成 DAG)。
- **Worker** — 注册自己的能力(tags / skills / 可处理项目等),被动接活。

## 需求 → 任务:拆解即任务（planning task 模式）

核心主张:**把"拆解"本身也变成一个任务,引擎只管派发,不内置"拆解"这个特殊能力。**

- 任务有两种来源,但对引擎完全一样:
  1. 直接在需求下创建的任务;
  2. 由一个 **planning task** 产出的子任务 —— 这个规划任务由人、或注册了 `planning`
     能力的 planner-agent(也是一种 worker)接走,其副作用是往需求里塞任务(可带 `depends_on`)。
- 流程:需求建好 → 引擎派一个 planning task → planner 接走 → 吐出带依赖的子任务
  → 子任务被正常 push 派发。引擎只在前置依赖满足时才派,"分析→设计→实现"的顺序天然成立。

为什么这个模型自然:
1. **引擎极简且统一** — 一切皆任务派发,"规划"只是"产出任务的任务",递归闭合,无第二套机制。
2. **可插拔** — 拆解是一种 worker 能力,不是内核;换 planner 引擎不动。
3. **同时吻合"任务集合容器"与"能被拆成任务"两种描述** — 拆解 = 往容器加任务的动作。
4. **和 Push 调度一致** — planner 就是注册了 `planning` 能力的 worker,走相同派发路径。

## 调度模型（Push + 能力匹配 + 依赖）

引擎(daemon)是"重"的一端,维护全部共享状态并主动调度:

1. 任务的 `depends_on` 前置全部完成 → 进入可调度状态。
2. 引擎按任务的**能力要求**匹配已注册、在线的 worker 能力。
3. 选定 worker,经其长连接 **push** 任务。
4. worker 执行并回报状态/产物;引擎更新共享状态,并经渠道回推。

worker 经 **WebSocket / SSE 长连接**注册并保持在线,引擎通过这条连接下发任务
(穿 NAT、实时性好、适合常驻 agent)。

## 进程 / 部署架构

```
              baton-daemon  (常驻服务器,引擎本体)
              共享状态 + 调度 + worker 长连接管理
                        ▲
                        │   一套完整 API（HTTP + WS）
        ┌───────────────┼────────────────┬──────────────┐
     CLI 端           Web 端          渠道(飞书/钉钉)    workers
   (功能完备)        (易读包装)        (双向)         (WS 订阅接活)
```

**核心原则:API-first / CLI-complete。**
daemon 的 API 是唯一契约;CLI 能跑通 100% 的功能(无头路径永远在);
Web 只是给人看的友好皮,绝不成为唯一入口;渠道与 worker 都是这套 API 的客户端。

**渠道双向**:既是入口(创建需求/任务、对任务操作),
也是出口(把进度、结果、需人确认的审批回推到原会话)。CLI 本身天然就是第一个渠道。

## 技术栈

- **TypeScript / Node**,全栈一种语言:daemon / CLI / web / worker-SDK 共享
  领域模型与 worker 协议的类型定义;和现有 trantor/tcollab 生态、函数式偏好一致。
- 遵循既有代码约定:`type` 优先于 `interface`、named export、不用 `any`、
  `import type`、不用 switch/case、文件 <200 行 / 函数 <40 行、偏函数式。
- monorepo(pnpm workspace):`daemon` / `cli` / `worker-sdk` / `shared`(领域类型+协议) / `web`(后做)。

## 模块路线图（一个个做的建议顺序）

> 每个模块动工前再出该模块的细化方案(`docs/plans/0X-*.md`);此处只定顺序与边界。

- **M1 · 引擎地基** — 领域模型(workspace/requirement/task/worker) + daemon 核心状态
  + HTTP API + CLI 跑通增删查。worker 先 stub,持久化先内存 + JSON 落盘。**(地基优先)**
- **M2 · worker 协议与接活** — WS 长连接、能力注册、push 派发、状态回报、
  依赖满足才派。一个最小本地 worker(echo/脚本)验证端到端闭环。
- **M3 · 真 agent worker** — 一个能跑 Claude Code 的 worker:接活 → 隔离工作目录执行
  → 回报结果/diff/日志。
- **★ 自举点(最少闭环达成后)** — M2/M3 把"渠道→引擎→worker 接活执行→回报"端到端跑通后,
  **用 baton 构建 baton**:baton 自身的开发任务进入 baton、由 worker 消费执行。
  这是"业务代码 + harness"方法论最强的兑现形式(见 `docs/` 之外的项目记忆)。
- **M4 · planning task / 需求拆解** — `planning` 能力的 worker,把需求拆成带依赖的子任务。
- **M5 · 渠道(双向)** — 飞书/钉钉适配器:创建 + 进度/结果/审批回推。
- **M6 · Web UI** — 同一套 API 的可读包装(Kanban / 需求树 / 任务详情)。

## 关键开放问题（留待对应模块动工时拍板）

- **持久化**:MVP 内存 + JSON 够用;何时上 SQLite / 真 DB?
- **worker 容错**:接活的 worker 挂了/超时,如何检测、重派、避免重复执行(幂等/租约)。
- **能力匹配语义**:tags 精确匹配,还是更结构化的能力描述 + 优选策略(负载/亲和性)?
- **人工介入点**:"审批"是任务的一种待人确认状态,经渠道回推 —— 状态机怎么表达。
- **隔离与工作目录**:worker 自己管 git worktree / 沙箱?引擎是否需要知道。
- **权限 / 多租户**:workspace 之间的隔离边界、谁能注册 worker、渠道身份如何映射到 workspace。
- **planning 产出的 DAG 表达**:子任务依赖如何序列化、如何回填到需求。
